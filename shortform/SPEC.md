# 숏폼 자동 생성·등록 파이프라인 스펙 (Codex 작업 지시서)

트렌드 주제 수집 → 원본 확보 → 하이라이트 탐지 → **번역·해설 대본 생성** →
숏폼 편집(컷/템포/줌인/화면전환/자막/TTS/에셋 애니메이션) → 렌더링 →
YouTube Shorts·Instagram Reels 자동 업로드까지 이어지는 24시간 파이프라인.

**핵심 콘텐츠 전략 — 해외 바이럴 번역·해설 쇼츠**: 해외에서 화제가 되고
있는(조회수 급상승) 영상·이슈를 찾아 한국어로 번역하고, 독창적인 해설
나레이션을 얹어 한국 시청자용 쇼츠로 재구성한다. 채널의 가치는 클립이
아니라 **해설(번역 + 맥락 + 관점 + 드립)**에 있다 — 해설이 주(主),
클립은 해설을 뒷받침하는 짧은 인용(종(從))이라는 원칙이 모든 모듈 설계에
우선한다.

---

## 0. 하드 제약 — 반드시 먼저 읽고 그대로 구현할 것

이 섹션은 기능 요구사항보다 우선한다. 위반하면 채널이 삭제되고 수익화가
불가능해지므로, "나중에 붙이는 옵션"이 아니라 파이프라인의 게이트로 구현한다.

### 0-1. 소스 권리 게이트 (rights gate)
- 소스 영상은 아래 화이트리스트 중 하나의 `license_type`이 DB에 기록된
  경우에만 편집 단계(M3 이후)로 진입할 수 있다. 미기록 소스는 상태를
  `BLOCKED_RIGHTS`로 전이시키고 파이프라인에서 제외한다.
  - `own` — 본인/자사 채널 원본
  - `licensed` — 서면 계약·허락을 받은 콘텐츠 (계약 문서 경로 필수 기록)
  - `cc_by` — Creative Commons 표시 라이선스 (출처 표기 자동 삽입 필수)
  - `public` — 퍼블릭 도메인, 공공누리 1유형 등 공공저작물
    (정치·시사 카테고리는 국회방송, 정부 브리핑 등 공공저작물 위주로 구성 가능)
  - `commentary` — 제3자 영상을 짧게 인용하며 독창적 해설을 얹는 변형적
    이용. **아래 0-1-1의 변형 규칙을 전부 통과해야만 유효**하며, 규칙은
    파이프라인이 기계적으로 검증한다.
- 아이돌 무대·뮤직비디오·예능·영화·방송사 클립은 `commentary` 모드로도
  다루지 않는다 (계약된 `licensed`만 가능). Content ID가 오디오·영상
  지문으로 즉시 매칭하는 고위험군이라, 해설을 얹어도 수익이 원저작자에게
  귀속되거나 차단되며 경고 3회면 채널이 삭제된다.
- YouTube 파트너 프로그램은 "재사용 콘텐츠(reused content)" 채널의 수익화를
  거부한다. 타인 영상을 자르거나 번역 자막만 단 채널은 조회수가 나와도
  수익이 0이다. 수익화의 근거는 해설·편집의 변형적 기여이며, 그 기여가
  실질적인지 아래 규칙으로 강제한다.

### 0-1-1. `commentary` 변형 규칙 (M4가 타임라인 검증 시 기계적으로 체크)
- **나레이션 비중**: TTS 해설이 러닝타임의 70% 이상을 커버할 것.
- **클립 인용 한도**: 제3자 클립은 개별 컷 8초 이하, 원본 푸티지 합계가
  러닝타임의 50% 이하일 것. 원본 오디오는 포인트 순간에만 살리고
  나머지는 -18dB 이하로 깔거나 제거.
- **원본 대체 불가성**: 쇼츠만 봐도 완결되되, 원본 영상의 핵심 가치를
  통째로 대체하지 않을 것 (예: 원본의 결말/전체 흐름 이관 금지).
- **출처 표기**: 크레딧 바(에셋 §6) + 설명란에 원본 채널·링크 자동 삽입.
- **소재 제한**: 음악(무대·MV·공연), 영화/드라마/방송 클립 금지.
  허용 소재: 해외 밈·챌린지, 해프닝/사건 영상, 테크·과학 데모, 스포츠
  일반 클립(리그 공식 중계 화면 제외), 인터뷰·강연 발췌 등 지문 매칭이
  약하고 인용 관행이 넓은 영역.
- **승인 큐 상시**: `commentary` job은 `AUTO_PUBLISH=true`여도 반드시
  운영자 승인을 거친다. 인용의 적절성 판단은 자동화할 수 없다.
- **한계 명시**: 위 규칙은 재사용 콘텐츠 정책 대응과 리스크 완화 장치이지
  법적 면책이 아니다. 소유권 주장(수익 귀속)·삭제 요청은 여전히 발생할 수
  있고, 발생 시 대응(이의제기/삭제/재편집)은 사람이 결정한다. M8이 클레임
  발생을 감지해 알림을 보내고 해당 소스 채널을 자동 블랙리스트에 올린다.

### 0-2. 발행 승인 게이트 (publish gate)
- 완전 무인 발행은 정책·품질 사고가 채널에 누적된 뒤에야 발견된다.
  기본값은 **승인 큐 모드**: 렌더링 완료본을 큐에 쌓고, 운영자가 웹 대시보드
  (M8)에서 승인한 건만 업로드한다.
- `AUTO_PUBLISH=true` 환경변수로 전환 가능하되, 이 경우에도
  정치·시사 카테고리는 항상 승인 큐를 거친다 (오정보·명예훼손 리스크).
- 하루 업로드 상한을 설정으로 강제한다 (기본 YouTube 3개 / Instagram 3개).

## 1. 결과물

- Python 3.12 모노리포. 저장소 루트에 `shortform/app/`로 구현.
- CLI 진입점 하나(`python -m app`)로 전체 파이프라인·개별 스테이지 실행 가능.
- systemd 또는 Docker Compose로 24시간 상주 가능한 스케줄러 포함.
- 승인/모니터링용 로컬 웹 대시보드 (FastAPI + 단일 HTML, 외부 CDN 금지).

## 3. 아키텍처 — 상태 머신 파이프라인

각 작업(job)은 SQLite `jobs` 테이블의 행이며 아래 상태를 순서대로 전이한다.
모든 스테이지는 멱등(idempotent)해야 하고, 실패 시 `retry_count`를 올리며
지수 백오프(2/4/8분, 3회)로 재시도, 초과 시 `FAILED`로 전이하고 알림.

```
DISCOVERED → RIGHTS_OK → ACQUIRED → ANALYZED → EDITED → RENDERED
   → (승인 큐) APPROVED → PUBLISHED_YT → PUBLISHED_IG → DONE
실패 분기: BLOCKED_RIGHTS / FAILED / REJECTED(운영자 반려)
```

디렉터리 구조:

```
shortform/
  SPEC.md               ← 이 문서
  ASSET_PROMPTS.md      ← ChatGPT 이미지 에셋 생성 프롬프트 모음
  app/
    __main__.py         # CLI: run-all | stage <name> | serve-dashboard
    config.py           # pydantic-settings, .env 로드
    db.py               # SQLite 스키마 + 상태 전이 함수
    stages/
      m1_discover.py
      m2_acquire.py
      m3_analyze.py
      m4_edit.py
      m5_render.py
      m6_publish.py
    editing/
      timeline.py       # 편집점 → 타임라인 JSON
      reframe.py        # 9:16 크롭 경로 계산
      subtitles.py      # ASS 생성
      tts.py
      ffmpeg_graph.py   # 타임라인 JSON → filtergraph 문자열
    dashboard/
      server.py         # FastAPI 승인 큐 + 상태 보드
      index.html
    scheduler.py
  assets/               # ChatGPT 생성 PNG (자막 박스, 스티커, 로고 등)
  work/                 # 다운로드·중간 산출물 (gitignore)
  out/                  # 최종 mp4 + 메타데이터 JSON (gitignore)
```

## 4. 모듈별 스펙

### M1 — 주제·소스 발굴 (`m1_discover`)
- 입력 소스 3종:
  1. **해외 트렌드 스캔(핵심)**: YouTube Data API `videos.list(chart=
     mostPopular)` regionCode `US|GB|JP` + 허용 카테고리, 그리고
     `search.list(order=viewCount, publishedAfter=48h)`로 키워드 스캔.
     각 후보에 대해 (a) **급상승 속도** = 조회수/게시경과시간, (b) **한국
     수용성 점수** = LLM 평가(한국 시청자에게 통할지, 문화 맥락 설명이
     필요한지 — 맥락 설명이 필요할수록 해설 가치가 높아 가점), (c) 0-1-1
     소재 제한 통과 여부를 산정해 우선순위를 매긴다.
  2. **소스 카탈로그**: `sources.yaml`에 운영자가 등록한 권리 확보
     채널/플레이리스트/로컬 파일 목록. 각 항목에 `license_type` 필수.
  3. **국내 트렌드 신호(보조)**: `mostPopular(regionCode=KR)` 키워드는
     제목·해시태그·해설 앵글 선정에만 사용.
- **중복 회피**: 같은 원본을 이미 다룬 한국 채널 수를 `search.list`로
  추정해 포화 소재는 감점, 대신 해설 앵글 차별화 힌트를 job에 기록.
- **후킹 벤치마크 수집**: 니치별로 조회수 상위 쇼츠(=알고리즘 검증이 끝난
  영상)를 주기적으로 수집해 `benchmarks` 테이블에 적재한다. 각 벤치마크의
  (a) 제목 패턴, (b) 첫 3초 훅 구도(대표 프레임 3장을 비전 LLM으로 분석:
  피사체 배치, 자막 위치·크기, 색 대비, 텍스트 훅 유형)를 구조화해
  `hook_patterns`로 저장 → M3 대본·제목 프롬프트와 M4 레이아웃 템플릿
  설계에 주입한다. 쇼츠는 피드에서 썸네일이 거의 노출되지 않으므로
  "썸네일"의 실체는 **첫 1~3초 프레임 구도**다 — 벤치마크 분석도 여기에
  집중한다. 구도·패턴만 참고하고 타인의 프레임·에셋을 복제하지 않는다.
- 출력: `jobs` 행 생성 (`DISCOVERED`), 카테고리(`global_meme|tech_science|
  sports|incident|politics|etc`), `license_type`(트렌드 스캔 발굴분은
  `commentary`), 우선순위 점수, 트렌드 키워드, 차별화 앵글 메모.

### M2 — 소스 확보 (`m2_acquire`)
- 권리 게이트 검증 후 `RIGHTS_OK` → yt-dlp(원격) 또는 파일 복사(로컬)로
  `work/{job_id}/source.mp4` 확보. 1080p 이상 우선, 오디오 분리 추출(wav).
- 메타데이터(원제, 채널, 길이, 라이선스 근거)를 job에 기록. `ACQUIRED`.

### M3 — 하이라이트 탐지 (`m3_analyze`)
세 신호를 합산해 편집점 후보를 만든 뒤 LLM으로 최종 선정한다.
1. **전사**: faster-whisper로 단어 단위 타임스탬프 전사.
2. **오디오 피처**: librosa RMS 곡선에서 상위 피크 구간(웃음·함성·강조 발화).
3. **장면 컷**: PySceneDetect 컷 리스트.
- 위 신호 + 전사문을 Claude에 넣고 카테고리별 프롬프트로 하이라이트
  구간(=인용할 클립 후보, `commentary`는 컷당 8초 이하 3~6개)을 JSON으로
  받는다. 프롬프트에 카테고리 규칙 포함:
  - `global_meme`: 밈의 핵심 순간(왜 터졌는지가 보이는 컷) 위주, 반복 컷 허용
  - `tech_science`: 데모의 before/after가 대비되는 컷, 수치·스펙 등장 순간
  - `sports`/`incident`: 결정적 순간 직전 1~2초 셋업 포함, 슬로모 재생 후보 표시
  - `politics`(공공저작물): 발언 맥락이 잘리지 않도록 문장 경계 스냅,
    발언자·날짜 자막 강제
- **댓글 마이닝**: 원본 영상의 댓글을 `commentThreads.list`(상위 좋아요순
  100개)로 수집해 LLM으로 구조화한다 — (a) 시청자들이 가장 반응한 순간/
  타임스탬프(하이라이트 선정 보정에 사용), (b) 반복되는 밈·유행 표현
  (해설 드립 재료), (c) 시청자가 궁금해하는 지점(해설이 답해줄 훅 후보),
  (d) 논란·민감 포인트(회피 목록). 결과는 `comment_insights`로 job에
  저장되어 제목·훅·대본 프롬프트에 주입된다.
- **후킹 제목 생성**: `hook_patterns`(M1 벤치마크) + `comment_insights`를
  근거로 제목 후보 5개를 생성하고, LLM 자체 평가(호기심 갭, 구체성,
  낚시성 과다 여부)로 상위 후보를 고른다. 과장·허위 낚시는 시청 지속률을
  망가뜨려 알고리즘에 역효과 — "궁금하게 하되 영상이 답을 주는" 제목만
  허용한다는 규칙을 프롬프트에 명시.
- **번역·해설 대본 생성 (`commentary` 핵심 단계)**: 전사문을 LLM으로
  한국어 의역(직역 금지 — 밈·슬랭은 한국어 감성으로 로컬라이즈)한 뒤,
  해설 대본을 생성한다. 대본 구조:
  1. **훅** (1문장, 1.5초 내): 시청자를 잡는 질문/반전 예고
  2. **맥락**: 이게 뭔지, 해외에서 왜 화제인지 (한국 시청자가 모르는 배경)
  3. **관전 포인트**: 클립 인용과 교차하며 짚어주는 디테일
  4. **코멘트/드립**: 채널 페르소나의 관점 한 방 (독창적 가치의 핵심)
  5. **마무리**: 한 줄 정리 + 다음 영상 유도
  - 채널 페르소나는 `persona.yaml`(말투, 1인칭, 금지 표현, 유머 톤)로
    관리하고 프롬프트에 주입 — 채널 전체의 목소리 일관성을 만든다.
  - 대본의 각 문단에 어떤 클립 컷을 붙일지 매핑까지 LLM이 출력한다.
- LLM 출력 스키마: `{clips: [{start, end, role}], script: [{text, clip_ref,
  emphasis_words[]}], hook_text, title, reason, keywords[]}`.
  클립 경계는 전사 타임스탬프의 문장 경계로 스냅한 뒤 저장. `ANALYZED`.

### M4 — 편집 결정 (`m4_edit`)
하이라이트·해설 대본을 "타임라인 JSON"으로 컴파일한다. 렌더와 분리해서
편집 로직을 단위 테스트 가능하게 유지한다. 컴파일은 M9의 템플릿 파라미터
(`layout`/`tempo`/`voice` 프리셋)를 입력으로 받아, 같은 대본으로 다른
변형 타임라인을 만들 수 있어야 한다.
- **`commentary` 타임라인 원칙**: TTS 나레이션 트랙이 뼈대다. 대본의
  `clip_ref` 매핑에 따라 나레이션 위에 클립을 배치하고, 클립이 없는
  구간은 정지 프레임+켄번즈 줌, 확대 리플레이, 슬로모, 에셋 오버레이로
  채운다. 컴파일 후 0-1-1 규칙(나레이션 ≥70%, 컷당 ≤8초, 푸티지 합계
  ≤50%)을 검증해 위반 시 자동으로 클립을 잘라내거나 `FAILED` 처리.
- **컷 편집·템포**: 무음(-35dB, 0.4초 이상) 자동 제거(jump cut).
  카테고리별 평균 샷 길이 목표: global_meme 1.5~2.5s, tech_science·sports
  2~3s, politics 4~6s(과편집 금지).
- **리프레임**: mediapipe 얼굴 박스의 EMA 스무딩 경로로 9:16 크롭.
  얼굴 미검출 구간은 중앙 크롭 + 상하 블러 패딩 폴백.
- **줌인/펀치인**: 강조 단어(LLM이 표시) 시점에 1.0→1.08 스케일 0.3초
  ease-out. 같은 샷 내 2회 이상 금지.
- **화면전환**: 기본은 하드컷. 섹션 경계에만 0.2초 crossfade 또는
  whip-pan(가로 블러) 중 택1. 3종 이상 섞지 말 것.
- **자막**: ASS로 워드 단위 팝인(현재 단어 색상 하이라이트, 폰트
  Pretendard Bold, 외곽선+그림자, 세이프존: 하단 25%·상단 12% 회피 —
  Shorts/Reels UI에 가리는 영역). 욕설 마스킹 테이블 적용.
- **TTS 나레이션**: 해설 대본 전체를 Edge-TTS로 문단 단위 생성(속도
  1.1~1.25배). `commentary`는 나레이션이 오디오 백본 — 원본 오디오는
  대본이 지정한 포인트 순간에만 살리고 나머지는 -18dB 이하 또는 제거.
  기타 타입은 훅+브릿지만 TTS, 원본 오디오를 -12dB 더킹(sidechaincompress).
  나레이션 문장별 실측 길이에 맞춰 타임라인 클립 길이를 재조정한다.
- **에셋 애니메이션**: `assets/` PNG 오버레이(스티커, 자막 박스, 채널 로고,
  구독 유도 배지)를 타임라인 이벤트로 배치. 등장은 0.25초 scale-bounce,
  퇴장은 0.15초 fade. 로고는 우상단 상시, 구독 배지는 마지막 2초.
- **아웃트로**: 마지막 1초 CTA 자막 + 다음 영상 유도 문구.
- 출력: `work/{job_id}/timeline.json`. `EDITED`.

### M5 — 렌더링 (`m5_render`)
- `timeline.json` → 단일 ffmpeg 명령의 filtergraph로 컴파일해 실행.
  (trim/concat, crop 키프레임은 `zoompan`/`crop` 표현식, overlay+fade,
  ASS burn-in, loudnorm I=-14 LUFS)
- 출력 규격: 1080x1920, H.264 high, CRF 18, 30fps(원본이 60이면 60 유지),
  AAC 192k. 길이 ≤ 60초(하드 제한 — Shorts는 3분까지 가능하지만 알고리즘
  안전값으로 60초 상한, 설정으로 변경 가능).
- 썸네일 프레임 1장 + 업로드 메타데이터 JSON(제목, 설명, 해시태그,
  CC-BY면 출처 표기 문단 자동 포함) 생성. `RENDERED`.

### M6 — 업로드 (`m6_publish`)
- **YouTube**: Data API v3 `videos.insert` (resumable). OAuth 리프레시 토큰
  파일 보관. 제목 끝 `#Shorts`. `selfDeclaredMadeForKids=false`.
  쿼터 주의: 업로드 1건 = 1,600유닛, 기본 일 쿼터 10,000 → 실질 6건/일.
- **Instagram Reels**: Graph API — 비즈니스 계정 필수. 영상은 공개 URL이어야
  하므로 `out/`을 임시 서명 URL로 서빙하는 업로드 헬퍼 포함
  (`media` 컨테이너 생성 → 상태 폴링 `FINISHED` → `media_publish`).
- 발행 시각: 카테고리별 슬롯(기본 07:30 / 12:30 / 19:30 KST)에 예약 분배.
- 결과 video ID·permalink를 job에 기록. `PUBLISHED_*` → `DONE`.

### M7 — 스케줄러 (`scheduler.py`)
- APScheduler 상주 프로세스: M1 매 2시간, M2~M5는 큐 워커(동시 렌더 1),
  M6은 발행 슬롯 cron. 프로세스 재시작 시 미완료 job 자동 복구(멱등성 활용).

### M8 — 대시보드·모니터링 (`dashboard/`)
- FastAPI: job 목록/상태, 렌더 결과 미리보기(video 태그), 승인/반려 버튼,
  실패 로그 뷰. 반려 시 사유 입력 → M3 프롬프트 개선에 쓸 수 있게 저장.
- 알림: `FAILED`·`BLOCKED_RIGHTS` 발생 시 웹훅(설정된 경우) POST.
- **클레임 감시**: 발행 영상의 Content ID 클레임·차단 상태를 일 1회 확인
  (업로드 계정 알림 메일 파싱 또는 Studio 수동 확인 체크리스트). 클레임
  발생 시 알림 + 해당 원본 채널을 `blacklist` 테이블에 추가해 M1이 재소싱
  하지 않게 한다.
- 성과 피드백: 발행 48시간 후 조회수·평균시청지속시간·시청완주율을
  YouTube Analytics API로 수집해 `performance` 테이블에 적재 → M1 우선순위
  가중치와 M9 최적화 엔진에 반영.

### M9 — 변형 실험·최적화 엔진 (`m9_optimize`)
같은 대본·제목이라도 레이아웃·디자인에 따라 성과가 갈리므로, 변형을
체계적으로 실험해 채널의 최적 폼(레이아웃·편집점·제목·톤앤매너)으로
수렴시킨다.
- **템플릿 차원**: 편집을 이산적인 차원의 조합으로 정의한다 —
  `layout`(자막 위치/크기/스타일, 훅 프레임 디자인, 에셋 배치),
  `tempo`(평균 샷 길이, 줌인 빈도), `voice`(TTS 보이스·속도),
  `title_formula`(질문형/반전형/수치형 등), `persona_tone`(드립 밀도,
  존댓말 여부). 각 차원의 프리셋은 `templates.yaml`로 관리하고 M4가
  타임라인 컴파일 시 파라미터로 받는다.
- **변형 생성**: job당 기본 2개(최대 3개) 변형을 서로 다른 템플릿 조합으로
  렌더링한다. 변형은 같은 플랫폼에 동시에 올리지 않는다 — 플랫폼 분산
  (YT/IG에 다른 변형) 또는 12시간 이상 시차 발행. 동일 콘텐츠 다변형을
  한 채널에 몰아 올리면 중복/스팸 판정 리스크가 있으므로 job당 동일 채널
  발행은 2변형까지로 제한.
- **밴딧 최적화**: 차원별 프리셋 성과를 Thompson sampling으로 학습한다.
  보상 지표 = 시청완주율(주지표) + 조회수 증가 속도(보조). 표본이 적은
  초기에는 탐색 비중을 높게 유지하고, 프리셋당 표본 10개 미만이면 결론을
  내리지 않는다 (쇼츠 성과는 분산이 커서 성급한 수렴이 최대 함정).
- **톤앤매너 수렴**: 주 1회 배치로 상위/하위 성과 영상의 대본·제목을
  LLM에 비교시켜 `persona.yaml`·제목 프롬프트 개선안을 생성하고, 운영자
  승인 후 반영한다 (프롬프트 자동 변경은 품질 드리프트 위험이 있어
  승인 게이트를 둔다). 승인 큐 반려 사유도 같은 배치의 입력으로 사용.
- 출력: `experiments`·`template_stats` 테이블, 대시보드에 차원별 승률
  리포트 노출.

## 5. 구현 순서 (Phase)

1. **Phase 1 — 편집 코어 (수동 입출력)**: 로컬 mp4 입력 → M3~M5 →
   `out/` 확인. 이 단계 완성도가 전체 품질을 결정하므로 가장 오래 투자.
2. **Phase 2 — 업로드**: M6 + 승인 대시보드. 수동 승인으로 실발행 검증.
3. **Phase 3 — 자동화**: M1·M2·M7 연결, 승인 큐 모드로 24시간 상주 운전.
4. **Phase 4 — 피드백 루프**: M8 성과 수집 + M9 변형 실험·밴딧 최적화 →
   레이아웃/편집점/제목/톤앤매너 수렴, 벤치마크·댓글 마이닝 고도화.

## 6. 검수 기준

- 로컬 10분 테스트 영상 1개 입력 시 사람 개입 없이 `out/`에 규격 준수
  mp4 + 메타데이터 JSON 생성.
- 자막 싱크 오차 ±0.2초 이내(샘플 20개 단어 수동 대조).
- 렌더 산출물에 검은 여백·세이프존 침범·오디오 클리핑 없음.
- `license_type` 없는 소스 투입 시 `BLOCKED_RIGHTS`로 차단되는 테스트 포함.
- 파이프라인 중간 kill 후 재시작 시 중복 산출물 없이 이어서 완료(멱등성 테스트).
- 시크릿(.env, 토큰 파일)은 gitignore. 저장소에 절대 커밋 금지.

## 7. 운영 메모

- 인스타 Reels API는 비즈니스 계정 전환 + Facebook 페이지 연결 + 앱 검수
  (`instagram_content_publish` 권한)가 선행돼야 한다. 검수 기간을 감안해
  Phase 2에서 YouTube 먼저 실발행하는 것을 권장.
- 수익화 관점 현실치: 이 채널의 상품은 클립이 아니라 해설이다. YPP 심사와
  재사용 콘텐츠 판정에서 살아남는 근거는 "번역 + 맥락 + 관점"이라는
  변형적 기여이므로, 해설 대본 품질(페르소나 일관성, 드립의 밀도)이
  파이프라인에서 가장 오래 튜닝해야 할 부분이다. 음악·영화·방송 클립을
  섞는 순간 채널 전체가 리스크에 노출된다는 점을 잊지 말 것. 승인 큐에서
  반려 사유를 축적해 대본 프롬프트를 계속 개선하는 운영 루틴을 권장.
